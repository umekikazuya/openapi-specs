# Contributing Guide

## コンポーネント追加の基準

以下を両方満たすもののみ追加する:

1. **2 プロジェクト以上で使われる** — 単一プロジェクト固有のコンポーネントはそのプロジェクトのリポジトリで管理する
2. **仕様が安定している** — 頻繁に変更が見込まれるものは安定してから追加する

## 破壊的変更の定義

以下のいずれかに該当する変更は**破壊的変更**とみなし、バージョンタグを上げる:

- フィールドの削除
- フィールドの型変更 (`string` → `integer` など)
- `required` への追加 (既存フィールドを必須化)

以下は**非破壊的変更**とみなし、同タグへの上書きで対応できる:

- フィールドの追加 (optional)
- `description` / `example` の修正
- フォーマット記述の改善 (意味が変わらない場合)

## タグの切り方

破壊的変更を加えたあとは新しいタグを作成して push する:

```bash
git tag v2
git push origin v2
```

各プロジェクトは URL 内のタグ番号を変更することでアップグレードする。

## PR の流れ

1. ブランチを作成して変更を加える
2. `spectral lint example/openapi.yaml` が通ることを確認する
3. `CHANGELOG.md` を更新する (Added / Changed / Removed いずれか)
4. 破壊的変更の場合はタグの更新準備が整っていることを確認する
5. PR を作成する — CI (Spectral lint) が自動実行される
